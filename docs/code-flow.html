smallnano — Complete Code Flow

From main() to a confirmed smn transfer — every function, every data transformation.
Language: Zig • Own protocol • Own network • Own coin

smallnano is its own cryptocurrency. It is inspired by Nano's block-lattice and weighted-voting ideas, but uses a completely independent wire protocol, address format (smn_…), and coin. It is NOT compatible with the Nano network. Fixed supply: 10,000,000 smn (10 million units, all minted in the genesis block).

1. Program Startup & Initialisation

The OS hands control to main(). No global state is initialised anywhere else. Every subsystem is constructed explicitly, in dependency order.

1
Entry Point
pub fn main() !void
src/main.zig
logic Allocates a std.heap.GeneralPurposeAllocator (GPA), calls Config.load(), constructs a Node, calls node.start(), then blocks waiting for SIGINT/SIGTERM before calling node.stop() and node.deinit(). The GPA reports any leaked allocations after deinit.
↓
pub fn Node.init(allocator: Allocator, config: NodeConfig) !Node
src/node.zig
logic Constructs all subsystems in dependency order — zero threads started here:
SqliteStore → Ledger → Network → VoteProcessor → ActiveElections → BootstrapClient → RpcServer → Wallet
↓
pub fn Node.start(self: *Node) !void
src/node.zig
infra Starts all I/O and background threads in order:
  1. Opens the SQLite database (SqliteStore.open())
  2. Runs pending schema migrations (SqliteStore.migrate())
  3. Verifies stored genesis hash matches genesis.zig — panics on mismatch
  4. Starts the TCP accept loop (Network.start())
  5. Starts the block processor worker thread
  6. Starts the vote processor thread
  7. Starts the election scheduler ticker
  8. Starts the RPC HTTP server
  9. If ledger is empty: inserts the genesis block and starts bootstrap

4. Network Layer Startup

4
TCP Listener + Peer Dialer
pub fn Network.start(self: *Network) !void
src/network/network.zig
infra Binds TCP on peering_port (default 7176). Spawns the accept loop thread. Loads known peers from the peers table and dials them. If the peer table is empty, resolves hardcoded bootstrap hostnames for the configured network (main / beta / dev).
↓
fn Network.accept_loop(self: *Network) void
src/network/network.zig
infra Blocks on accept(). Wraps each connection in a Channel, spawns a read-loop thread (bounded by max_peers). Excess connections are rejected immediately.

5. Peer Handshake

Every TCP connection — inbound or outbound — begins with a Node-ID handshake before any other messages are allowed. This authenticates the remote peer's identity and binds the session to their Ed25519 public key.

5
Cookie / Response handshake
pub fn Handshake.run(channel: *Channel, our_node_id: KeyPair) !PeerId
src/network/handshake.zig
network
  1. Generates a 32-byte random cookie via std.crypto.random.bytes().
  2. Sends Handshake { magic: 0x534E, type: .Query, cookie }.
  3. Receives peer's Handshake { type: .Response, node_id_pubkey, sig }.
  4. Calls Ed25519.verify(cookie, sig, node_id_pubkey) — drops connection on failure.
  5. Signs our cookie with our_node_id.secret_key, sends our .Response.
  6. Returns authenticated PeerId { pubkey, address }.
↓
pub fn Ed25519.verify(msg: []const u8, sig: [64]u8, pubkey: [32]u8) !void
src/crypto/ed25519.zig
crypto Wraps std.crypto.sign.Ed25519.verify(). Returns error.InvalidSignature on failure.

6. Receiving a Published Block

A peer broadcasts a new block as a Publish message. The channel read loop deserialises it and routes it to the block processor queue.

6
Deserialise → Block Processor Queue
fn Channel.read_loop(self: *Channel) void
src/network/channel.zig
infra Reads the 8-byte smallnano message header (magic u16 | network u8 | version u8 | type u8 | _ u8 | body_len u32). Reads body_len bytes into an arena-allocated buffer. Dispatches to Message.deserialize(type, body). Bans the peer for the session on error.InvalidMagic or unknown type.
↓
pub fn Message.deserialize(msg_type: MessageType, body: []const u8) !Message
src/network/message.zig
logic For .Publish: reads 216 bytes → StateBlock (account 32 + previous 32 + representative 32 + balance 16 + link 32 + work 8 + sig 64). All integers little-endian. Returns error.UnexpectedEof on truncation.
↓
pub fn BlockProcessor.submit(self: *BlockProcessor, block: StateBlock, source: BlockSource) void
src/ledger/block_processor.zig
logic Pushes the block onto a bounded MPSC queue (capacity 4096). The worker thread drains it by calling Ledger.process(block) for each entry. If the queue is full, the block is dropped and a stat.dropped_blocks counter incremented.

7. Block Validation (pure logic)

Validation is entirely pure — no disk I/O inside the validator. The ledger reads all required account state before calling the validator, passes it as plain data, then acts on the result. This is the Logic Sandwich from A-frame architecture.

7
BlockValidator
pub fn BlockValidator.validate(block: StateBlock, ctx: ValidationContext) BlockError!void
src/ledger/validator.zig
logic Runs all rules in order, fails fast on first violation:
RuleError
Ed25519 signature over block hash is validerror.BadSignature
PoW meets difficulty threshold for block typeerror.InsufficientWork
Previous block exists (or zero-hash for account open)error.GapPrevious
Block hash not already in ledgererror.Old
Balance does not go negativeerror.NegativeSpend
No competing block on same previous (fork)error.Fork
Pending entry exists and unreceived (receive blocks)error.Unreceivable
Not the burn account (all-zeros pubkey)error.OpenedBurnAccount
ValidationContext is a plain struct: the validator takes no allocator, makes no syscalls, and is trivially unit-testable.
↓
pub fn Blake2b.block_hash(block: StateBlock) [32]u8
src/crypto/blake2b.zig
crypto Computes the block hash as Blake2b-256 over a fixed 176-byte input:
preamble(0x06) | account(32) | previous(32) | representative(32) | balance(16) | link(32)
(work and signature are NOT included in the hash — they are over/about it)
↓
pub fn Work.is_valid(work: u64, hash: [32]u8, threshold: u64) bool
src/crypto/work.zig
crypto Computes Blake2b-64(work_le_bytes ++ hash). Returns true if the 64-bit result ≥ threshold (little-endian u64). Send/change blocks use a harder threshold than receive/open blocks.

8. Block Insertion into the Ledger

8
Ledger.process → BlockInserter → SQLite writes
pub fn Ledger.process(self: *Ledger, block: StateBlock) !ProcessResult
src/ledger/ledger.zig
logic The Logic Sandwich:
  1. Reads AccountInfo, previous block, pending entry from the store (infra).
  2. Builds ValidationContext from that data.
  3. Calls BlockValidator.validate(block, ctx) (pure logic).
  4. On success: calls BlockInserter.insert(block, ctx) (infra).
  5. Calls LedgerPruner.maybe_prune(block.account).
  6. Emits LedgerEvent.BlockProcessed to subscribers.
  7. Returns .Progress (block inserted) or .Gap (missing previous).
↓
pub fn BlockInserter.insert(self: *BlockInserter, block: StateBlock, ctx: InsertContext) !void
src/ledger/inserter.zig
infra All writes in a single SQLite transaction (BEGIN IMMEDIATE … COMMIT):
  • INSERT INTO blocks (hash, account, block, height)
  • INSERT OR REPLACE INTO accounts (account, frontier, balance, representative, height, modified)
  • If send block: INSERT INTO pending (hash, account, amount, source)
  • If receive block: DELETE FROM pending WHERE hash = ?
  • Updates in-memory representative weight cache
Atomicity: a crash mid-write leaves the DB in the pre-insert state (SQLite WAL guarantee).

9. Election & Weighted Voting

After insertion, the block is scheduled for an election. Representatives (high-balance accounts with enable_voting = true) broadcast votes. The election tallies weighted votes until quorum is reached.

9
Election state machine (pure logic)
pub fn ActiveElections.start_election(self: *ActiveElections, block: StateBlock) !void
src/consensus/active_elections.zig
logic Creates an Election keyed by (account, previous_hash) — the root. If an election already exists for this root, the new block is registered as a fork candidate. If the container exceeds max_pending_elections, the election with the lowest total tallied weight is evicted.
↓
pub fn VoteProcessor.process(self: *VoteProcessor, vote: Vote) void
src/consensus/vote_processor.zig
logic
  1. Verifies Ed25519 signature over Blake2b256(hash_list_bytes ++ timestamp_le_bytes).
  2. Checks the (representative, timestamp) pair against a dedup cache.
  3. Looks up the representative's confirmed balance from the weight cache.
  4. Calls Election.add_vote(rep, weight, block_hash) for each hash in the vote.
↓
pub fn Election.add_vote(self: *Election, rep: [32]u8, weight: u128, block_hash: [32]u8) ElectionStatus
src/consensus/election.zig
logic Accumulates vote weight per candidate block. Quorum check:
tallied * 3 >= online_weight * 2 (pure integer, no floats, no division)
Returns one of:
  • .ongoing — quorum not yet reached
  • .confirmed { winner: [32]u8 } — winner has ≥ 67% of online weight
  • .fork — two candidates each have significant weight

10. Confirmation & Cementation

10
ConfirmationTracker → SQLite cementation write
pub fn ConfirmationTracker.on_confirmed(self: *ConfirmationTracker, winner_hash: [32]u8) !void
src/consensus/confirmation.zig
logic Walks backward through the winning block's chain from winner_hash down to the current confirmation_height for that account. Cements each block (height must be monotonically increasing — never goes backward).
↓
fn SqliteStore.write_confirmation_height(account: [32]u8, height: u64, frontier: [32]u8) !void
src/store/sqlite_store.zig
infra INSERT OR REPLACE INTO confirmation_height (account, height, frontier) VALUES (?, ?, ?)
This write is the canonical "transfer is finalised" moment. Once this row exists, the smn have irreversibly moved to the recipient's pending entry. The send block is now spendable by the recipient.

11. Ledger Pruning

After each insertion, the pruner enforces the max_blocks_per_account limit. This is what makes smallnano runnable on cheap hardware. Pruning never removes blocks at or below the confirmation watermark — confirmed state is always preserved.

11
LedgerPruner
pub fn LedgerPruner.maybe_prune(self: *LedgerPruner, account: [32]u8) !void
src/ledger/pruner.zig
logic Reads block count and confirmation height for the account.
If block_count - confirmed_height > max_blocks_per_account:
Computes prune_up_to = confirmed_height - max_blocks_per_account
Calls store.delete_blocks_below(account, prune_up_to)
Writes new watermark to the pruned table.
↓
fn SqliteStore.delete_blocks_below(account: [32]u8, height: u64) !u64
src/store/sqlite_store.zig
infra DELETE FROM blocks WHERE account = ? AND height < ?
Returns number of rows deleted. Runs in the same transaction as the watermark write.
Pruned node telemetry: Nodes advertise their pruning watermark in Telemetry messages. Peers that need deeper history skip pruned nodes for bootstrap and pick archival nodes instead.

12. Initiating a Transfer (send side)

A user wants to send smn to another account. The flow starts from the wallet or the RPC send command.

12
Wallet → Build Send Block → CPU PoW → Sign → Broadcast
pub fn Wallet.create_send(self: *Wallet, from: [32]u8, to: [32]u8, amount: u128) !StateBlock
src/wallet/wallet.zig
logic
  1. Reads AccountInfo for from: frontier, balance, representative.
  2. Checks balance >= amount — returns error.InsufficientFunds otherwise.
  3. Builds unsigned StateBlock:
    { account: from, previous: frontier, representative: rep, balance: balance - amount, link: to }
  4. Computes block hash: Blake2b.block_hash(block)
  5. Generates PoW: Work.generate(hash, send_threshold, config.work_threads)
  6. Signs: Ed25519.sign(hash, secret_key)
  7. Returns the complete, signed, work-stamped StateBlock.
↓
pub fn Work.generate(hash: [32]u8, threshold: u64, threads: u8) u64
src/crypto/work.zig
crypto CPU brute-force using threads workers (each tries random nonces). Computes Blake2b-64(nonce_le ++ hash) until result ≥ threshold. Workers share a std.atomic.Value(bool) "found" flag; the winner stores the nonce atomically. ~2–5 seconds on a Raspberry Pi Zero 2 for a send block.
↓
pub fn Network.broadcast(self: *Network, block: StateBlock) void
src/network/network.zig
network Serialises the block into a Publish message and enqueues it on every active channel's write queue. Also submits the block to the local BlockProcessor so it is inserted into the local ledger immediately.

13. Completing a Transfer (receive side)

The recipient's node has the send block confirmed and sees a pending entry. The wallet builds a receive block to claim the smn.

13
Wallet → Build Receive Block → CPU PoW → Sign → Broadcast
pub fn Wallet.create_receive(self: *Wallet, account: [32]u8, pending_hash: [32]u8) !StateBlock
src/wallet/wallet.zig
logic
  1. Reads the pending entry: store.get_pending(account, pending_hash) → PendingInfo { source, amount }.
  2. Reads AccountInfo (or zero values if account never opened).
  3. Builds StateBlock { account, previous: frontier, representative: rep, balance: balance + pending.amount, link: pending_hash }.
  4. Computes block hash, generates PoW (lighter receive_threshold), signs.
  5. Returns the complete receive block.
↓

The receive block then flows through the same path as any published block:
Network.broadcast → BlockProcessor.submit → Ledger.process → BlockValidator → BlockInserter → ActiveElections → VoteProcessor → Election.add_vote → ConfirmationTracker → SQLite cementation.
When confirmation height is written for the receive block — the transfer is complete and irreversible.

14. RPC API Layer

14
HTTP JSON-RPC (port 7177)
pub fn RpcServer.accept_loop(self: *RpcServer) void
src/rpc/server.zig
infra Single-threaded HTTP/1.1 server. Reads request line + headers + JSON body. Deserialises the "action" field. Dispatches to the matching handler. Serialises the JSON response and writes it back. All handlers receive a pointer to Node and read from the ledger/network directly.
Example — process action:
Client POSTs {"action":"process","block":{"type":"state","account":"smn_...","previous":"...","balance":"...","link":"...","work":"...","signature":"..."}}.
Handler calls StateBlock.from_json(body, arena), then ledger.process(block). Returns {"hash":"<hex>"} on success or {"error":"<reason>"}.

15. Bootstrap Sync

On first start (or after being offline), the node syncs missing blocks from peers using the smallnano PullReq / PullAck messages.

15
BootstrapClient
pub fn BootstrapClient.run(self: *BootstrapClient) !void
src/bootstrap/client.zig
network
  1. Sends a frontier request to a bootstrap peer — receives all account frontiers the peer holds.
  2. For each account where peer's frontier ≠ local frontier: sends PullReq { account, start_hash, count: 128 }.
  3. Receives PullAck { blocks: []StateBlock }.
  4. Each block is submitted to BlockProcessor.submit(block, .Bootstrap).
  5. Only requests down to max_blocks_per_account blocks from tip — never requests pruned history.
  6. Stores checkpoint (last processed account + hash) in SQLite; resumes after restart.
  7. Rate-limited via bandwidth_limit_mbps token bucket.

16. Graceful Shutdown

16
SIGINT / SIGTERM → stop → deinit
pub fn Node.stop(self: *Node) void
src/node.zig
infra Sets the shared CancellationToken. All polling loops check it on each iteration and exit. Joins all background threads in reverse startup order. Flushes the SQLite WAL: PRAGMA wal_checkpoint(FULL).
↓
pub fn Node.deinit(self: *Node) void
src/node.zig
infra Calls deinit() on every subsystem in reverse construction order. SqliteStore.deinit() closes the DB connection. The GPA in main() then reports any remaining allocations — in a correct run, this number is zero.
